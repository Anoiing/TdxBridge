# TdxBridge 接口与联调参考

这个参考文件用于补充 `tdxbridge-agent` 的具体调用顺序、请求体和风险边界。

## 1. 推荐联调顺序

按下面顺序做，不要跳步：

0. 先确认 Windows 用户已登录桌面，且 `TdxW.exe` 已启动
1. `GET /health`
2. `GET /doc`
3. `GET /capabilities`
4. `POST /stock/search`
5. `POST /rpc/python` with `get_user_sector`
6. `POST /rpc/python` with `get_stock_list`

如果第 1 步或第 2 步都不通，先排查服务地址、Token、Windows 登录状态、`TdxW.exe`、`127.0.0.1:17709`、白名单和防火墙。

如果 `TdxW.exe` 没启动，先启动通达信客户端，再继续后面的 HTTP 联调。

如果要长期稳定接入，还要补查：

1. 通达信客户端是否配置了开机自启
2. 安装摘要里是否打印 `通达信客户端开机自启：已开启`
3. 如果没有配置开机自启，明确告知用户：当前实例只适合“人工先登录并启动通达信后再用”

## 2. 默认认证方式

除非实例文档明确说明匿名可访问，否则默认带：

```http
Authorization: Bearer <token>
Content-Type: application/json
```

## 3. 低副作用请求示例

### `GET /health`

用途：确认服务启动状态，并查看健康检查项。

### `GET /capabilities`

用途：确认实例支持的接口、运行模式、白名单和 warning。

关键检查点：

- `pythonRuntime.mode` 应优先为 `subprocess-per-call`
- warning 里是否提示缺少目录、占位 token 或来源限制

### `GET /doc`

用途：查看当前 bridge 的完整接口总表、参数入口、底层 method 映射、统一响应结构和风险边界。

### `POST /stock/search`

用途：验证本地通达信读取通路。

```json
{"keyword":"寒武纪"}
```

### `POST /rpc/python`

用途：验证 Python 子进程隔离通路。

请求体 1：

```json
{"method":"get_user_sector","params":{}}
```

请求体 2：

```json
{"method":"get_stock_list","params":{}}
```

注意：`get_stock_list` 结果可能很大，输出摘要、数量或前几项即可，不要整份展开。

- `GET /health`
- `GET /doc`
- `GET /capabilities`
- `POST /rpc/auto`
- `POST /rpc/local`

- `POST /formula/runBatch`

## 5. 高风险接口

在没有用户明确授权前，不要调用：

- `POST /trade/order`
- `POST /trade/cancel`
- `POST /warn/send`
- `POST /backtest/send`
- `POST /blocks/push`

也不要调用：

- `order_stock`
- `cancel_order_stock`
- `send_warn`
- `send_bt_data`

## 6. 排障顺序

1. 服务地址是否正确
2. Token 是否正确
3. `/health` 是否可达
4. `TdxW.exe` 是否运行
5. 通达信客户端开机自启是否已经配置
6. `127.0.0.1:17709` 是否可达
7. `/capabilities` 中 `pythonRuntime.mode` 是否正常
8. 白名单和防火墙是否限制了当前来源
9. `logs/service.stdout.log` 与 `logs/service.stderr.log` 是否有明显报错
